## CRITICAL (8 findings: 6 PREV + 2 NEW v2)

### C1. [PREV] `d1_truss_elementstrain` sign error — returns wrong sign for strain

- **File:** `src/truss.jl:73-76`
- **Found by:** Prosecutor 🔴
- **Formula:** `return 1 / L * [1 -1; -1 1] * u`
- **Problem:** Computes `(u₁−u₂)/L` for the first component. Physical engineering strain is `(u₂−u₁)/L = ΔL/L`. Sign is flipped.
- **Test encodes wrong sign:** `test/runtests.jl:111` — `@test eps ≈ [2.5e-4; -2.5e-4]` matches the wrong sign.
- **Cross-validator:** ✅ Still valid — no changes to `truss.jl`
- **Fix:** Replace matrix with `[-1 1; 1 -1]` or compute `1/L * [-u₁ + u₂; u₁ - u₂]`.

### C2. [PREV] 3D angle semantics: direction cosines documented as "Euler angles" — silently wrong for new users

- **File:** `src/spring.jl:140-153`, `src/truss.jl:280-294`
- **Found by:** Defender 🛡️, Prosecutor 🔴
- **Problem:** Parameters `thetax, thetay, thetaz` documented as "Angle about x-axis in degrees" but are actually direction cosines. A thinking Euler angles gets silently wrong results.
- **No check** that cos²αx + cos²αy + cos²αz ≈ 1. Test uses (0,0,0) giving sum=3 — physically impossible.
- **Cross-validator:** ✅ Still valid — docstrings unchanged, no validation added
- **Fix:** Rename parameters to `αx, αy, αz` (angles from coordinate axes). Validate `sum(C²) ≈ 1`.

### C3. [PREV] Near-vertical 3D beam: exact equality check bypasses special case, causes catastrophic precision loss

- **File:** `src/beam.jl:204,301`
- **Found by:** Prosecutor 🔴, Defender 🛡️
- **Problem:** `if x1 == x2 && y1 == y2` triggers special case only for EXACT bitwise equality. A beam `(0,0,0)→(1e-14,1e-14,4)` falls through to general branch where `D = √(Cx²+Cy²) ≈ 3.5e-15`, dividing by it amplifies noise 2.8e14×.
- **Cross-validator:** ✅ Still valid — no changes to `beam.jl`
- **Fix:** Use threshold `hypot(Cx, Cy) < 1e-12`. Extract `_d3_beam_rotation_matrix()` helper.

### C4. [PREV] `d2_beam_elementforces` returns LOCAL forces; `d2_beam_elementstiffness` returns GLOBAL matrix — undocumented

- **File:** `src/beam.jl:52-59` vs `:105`
- **Found by:** Prosecutor 🔴, Defender 🛡️
- **Problem:** Stiffness matrix is global, forces computed as `kprime * T * u` (local). No docstring warns. Users comparing `K_global * U` with `elementforces` get different values.
- **Cross-validator:** ✅ Still valid — no docstring updates
- **Fix:** Document that element forces are in LOCAL coordinates in both docstrings and README.

### C5. [PREV] API inconsistency across dimension variants: d1 returns vectors, d2/d3 return scalars

- **File:** `src/truss.jl:40-42` vs `:161-168`, `:313-322`; `src/spring.jl:32-34` vs `:96-101`
- **Found by:** Prosecutor 🔴, Human Factors 👤, Defender 🛡️

### C7. [NEW v2] d2/d3 diagram sign convention wrong — `plot_diagram` uses cos(θ) projection, MATLAB uses −sin(θ)

- **File:** `src/plot.jl:25-202` (all 8 diagram functions)
- **Found by:** Prosecutor 🔴
- **Problem:** The generic `plot_diagram` pattern computes `diagram_value[i] = scale * cos(θ)` (top-projection of the force onto the element). The MATLAB reference at `Doc/Kattan/M-Files/BeamElementAxialDiagram.m` uses `-scale * sin(θ)` (right-projection). This means every beam diagram shows the force component projected onto the wrong axis — visually the diagram shape is physically wrong.
- **Impact:** A student comparing LibFEM beam diagrams against textbook/MATLAB output sees different shapes and assumes the Julia implementation is buggy. This is a correctness bug in the visualization layer.
- **Test coverage:** Diagram tests (`runtests.jl:191+`) only check `p isa Plots.Plot` — never verify the plotted values.
- **Fix:** Change the projection from `cos(θ)` to `−sin(θ)` matching MATLAB. Add structural diagram tests that check plotted y-values. Extract shared `_project_force(f, L, θ)` helper.

### C8. [NEW v2] `_assemble!` silently skips out-of-bounds DOF indices instead of erroring

- **File:** `src/assembly.jl:21-28`
- **Found by:** Defender 🛡️, Human Factors 👤
- **Problem:** `_assemble!` uses `K[nodelist, nodelist] .+= ke` where `nodelist` is computed from `(i-1)*ndofs+1 : i*ndofs`. If `i=0` or `i*ndofs > size(K,1)`, the view creation silently truncates or wraps rather than erroring. Example: `d1_spring_assemble(K, ke, 0, 1)` writes to indices `0:0` (empty slice) — no error, no assembly, silent wrong result.
- **Worse:** Since Julia 1.11+ 0-based indexing in views may change behavior further. In current Julia, `0:0` is treated as empty range — the element is silently lost.
- **Cross-reference:** Related to H2 (size check) but distinct — this is about **index validity**, not matrix size.
- **Fix:** Add `1 ≤ i ≤ size(K,1)/ndofs` check (and `j` equivalent) before indexing. Throw `AssemblyError` with descriptive message.

### H8. [NEW v2] README claims "A > 0" validation exists — source code has NO validation for A ≤ 0

- **File:** `README.md` (Conventions section: "Most stiffness/length functions validate positive inputs (`L > 0`, `A > 0`)")
- **Found by:** Cross-Validator ✅
- **Evidence:**
  - `README.md` explicitly states: *"Most stiffness/length functions validate positive inputs (`L > 0`, `A > 0`)"*
  - Grep of all source: `L > 0 || throw(...)` appears in beam.jl and truss.jl at 5+ sites
  - Grep of `A > 0`: **zero matches** in any source file
  - `truss.jl:39` `d1_truss_elementstiffness(E, A, L)` takes `A` as a parameter but does not validate `A > 0`
  - `truss.jl:141` `d2_truss_elementstiffness(E, A, L, theta)` — same, no `A` check
- **Impact:** New users trusting the README assume `A ≤ 0` is caught. A zero/negative cross-section silently produces a zero/negative stiffness — physically wrong but no error.
- **Note:** This was found during cross-validation of PREV finding H1. H1 focused on `E ≤ 0`/`G ≤ 0`/`k ≤ 0`; this is specifically about the **README's false claim** of existing `A > 0` validation. The code validates `L > 0` but NOT `A > 0` despite claiming both.

### H9. [NEW v2] d3_beam_elementstiffness no sanity check on G, J, Iy, Iz — if G=0, singular NaN matrix silently

- **File:** `src/beam.jl:169-228` (d3_beam_elementstiffness)
- **Found by:** Prosecutor 🔴
- **Problem:** `d3_beam_elementstiffness(E, G, A, Iy, Iz, J, x1, y1, z1, x2, y2, z2)` constructs a 12×12 matrix. If `G=0.0` (shear modulus not provided or zero), the torsional stiffness terms vanish and the matrix becomes singular. Similarly `Iy=0` or `Iz=0` produces rank-deficient matrices. In the current code, these produce no warning — the assembly runs, the solver runs, and the user gets silently wrong displacements.
- **Worse:** The `@kwdef Beam{3}()` struct defaults `E=0, G=0, A=0, Iy=0, Iz=0, J=0` — a user creating a Beam and forgetting to set `G` gets a singular system.
- **Test coverage:** `runtests.jl:338-366` uses only one set of valid parameters for d3_beam. No test exercises edge cases (`G=0`, `Iy=0`, etc.).
- **Fix:** Validate `G > 0`, `Iy > 0`, `Iz > 0`, `J ≥ 0` (torsional constant can be zero for some sections, but shear modulus `G` must be > 0). At minimum, warn for zero values.

### H10. [NEW v2] Sign convention completely undocumented — no docstring states whether positive = tension or compression

- **File:** All `*_elementforce`, `*_elementstress`, `*_elementstrain` docstrings across `src/spring.jl`, `src/truss.jl`, `src/beam.jl`
- **Found by:** Human Factors 👤
- **Problem:** Not a single function docstring documents the sign convention. A positive return value from `d1_truss_elementforces` could be tension OR compression depending on convention — the docstring says "nodal force vector" with no sign convention. The MATLAB reference uses structural engineering convention (tension positive), but nothing in LibFEM confirms this.
- **Worse:** C1 (strain sign error) shows the implementation may not even be internally consistent. Without documented sign convention, users cannot know whether the sign is correct.
- **Impact:** Critical for an educational library. Students modeling a truss and getting negative element forces don't know if it's compression or a bug.
- **Fix:** Add to every `*_elementstrain`, `*_elementstress`, and `*_elementforce` docstring: **"Returns [quantity]. Positive value indicates tension, negative indicates compression (structural engineering sign convention)."**

## Key Takeaways

1. **Zero progress since v1 review.** All 36 previous findings remain unfixed. The codebase has not changed between reviews.
2. **Most impactful NEW finding: C7 — diagram sign convention wrong.** The d2/d3 beam diagrams use a different projection axis than the MATLAB reference, producing visually wrong diagrams. This is a correctness bug in the visualization layer that every student using diagrams would encounter.
3. **Second most impactful NEW finding: C8 — _assemble! silently skips out-of-bounds indices.** No error when `i=0` or index exceeds matrix size — element is silently lost. This is a correctness bug that can produce wrong results with no warning.
4. **Documentation integrity issue: H8 — README claims A>0 validation exists but code does not validate A.** This erodes trust in the documentation's accuracy.
5. **Pedagogical gap: H10 — sign convention completely undocumented.** For an educational library, this is a significant usability failure. Students don't know if positive force = tension or compression.
6. **Pattern duplication still the dominant code quality issue.** C7 (diagram), M15 (d2 beam rotation), H3 (d3 beam rotation), H4 (direction cosines), M5 (diagram functions) — all are variants of copy-paste repetition. The repeated finding across both reviews indicates a systematic lack of refactoring discipline.
