## CRITICAL (6 findings)

### C1. [NEW] `d1_truss_elementstrain` sign error — returns wrong sign for strain

- **File:** `src/truss.jl:73-76`
- **Found by:** Prosecutor 🔴
- **Formula:** `return 1 / L * [1 -1; -1 1] * u`
- **Problem:** Computes `(u₁−u₂)/L` for the first component. Physical engineering strain is `(u₂−u₁)/L = ΔL/L`. Sign is flipped.
- **Test encodes wrong sign:** `test/runtests.jl:111` — `@test eps ≈ [2.5e-4; -2.5e-4]` matches the wrong sign.
- **Blast radius:** All users computing 1D truss/bar strain get wrong sign. Stress uses same formula path.
- **Fix:** Replace matrix with `[-1 1; 1 -1]` or compute `1/L * [-u₁ + u₂; u₁ - u₂]`.

### C2. [NEW] 3D angle semantics: direction cosines documented as "Euler angles" — silently wrong for new users

- **File:** `src/spring.jl:140-153`, `src/truss.jl:280-294`
- **Found by:** Defender (Blindspot #4), Prosecutor (#4)
- **Problem:** Functions like `d3_spring_elementstiffness(k, thetax, thetay, thetaz)` document parameters as "Angle about x-axis in degrees" etc., but they are actually **direction cosines** — `cos(thetax)` = cosine of angle between member axis and x-axis. A user thinking Euler angles (α, β, γ) gets silently wrong results.
- **Validation missing:** No check that `cos²(thetax) + cos²(thetay) + cos²(thetaz) ≈ 1`. Test at `runtests.jl:395` uses `(0,0,0)` giving Cx²+Cy²+Cz²=3 — physically impossible.
- **Fix:** Rename parameters to `αx`, `αy`, `αz` (angles from coordinate axes) with clear docstrings. Optionally validate `sum(C²) ≈ 1`.

### C3. [NEW] Near-vertical 3D beam: exact equality check bypasses special case, causes catastrophic precision loss

- **File:** `src/beam.jl:204,301`
- **Found by:** Prosecutor (#3), Defender (#5, #15)
- **Problem:** `if x1 == x2 && y1 == y2` triggers special case only for EXACT bitwise equality. A beam with coordinates `(0,0,0)→(1e-14, 1e-14, 4)` falls through to general branch where `D = sqrt(Cx²+Cy²) ≈ 3.5e-15`, dividing by it amplifies noise 2.8e14×.
- **Fix:** Use threshold `hypot(Cx, Cy) < 1e-12` instead of `x1 == x2 && y1 == y2`. Extract rotation matrix construction into shared `_d3_beam_rotation_matrix()` helper.

### C4. [NEW] `d2_beam_elementforces` returns LOCAL forces; `d2_beam_elementstiffness` returns GLOBAL matrix — undocumened

- **File:** `src/beam.jl:52-59` vs `:105`
- **Found by:** Prosecutor (#6), Defender (#6)
- **Problem:** Stiffness matrix is in global coordinates (users assemble `K_global`). Forces are computed as `kprime * T * u` (local frame). Same for `d3_beam`. No docstring warns of this mismatch.
- **Impact:** Users comparing `K_global * U` (global reactions) with `elementforces(...)` get different values and assume bugs. This is correct FEA post-processing (forces are naturally local), but the inconsistency is hidden.
- **Fix:** Document that element forces are in LOCAL coordinates. Add note to both function docstrings and the README.

### C5. [NEW] API inconsistency across dimension variants: d1 returns vectors, d2/d3 return scalars

- **File:** `src/truss.jl:40-42` vs `:161-168`, `:313-322`; also `src/spring.jl:32-34` vs `:96-101`
- **Found by:** Prosecutor (#2, #10), Human Factors (#1), Defender (#7)
- **Problem:** 
  - `d1_truss_elementstrain` returns 2-element vector; `d2/d3_truss_elementstrain` return 1-element vector (scalar)
  - `d1_truss_elementstress(Ke, u, A)` takes stiffness matrix; `d2_truss_elementstress(E, L, theta, u)` takes raw params
  - `d1_spring_elementforce(Ke, u)` returns nodal forces (2-vec); `d2_spring_elementforce(k, theta, u)` returns scalar axial force
- **Fix:** Pick ONE convention per operation family and migrate. Discuss: d1 should return per-node values (as now), or all should return scalars with consistent signatures.

### C6. [PREV] Type hierarchy is decorative — structs never used by computation functions

- **File:** `src/types.jl` (full), **Found by:** All 5 agents
- **Previous #2:** $8\times8$ structs/aliases/show methods; zero functions dispatch on them. All functions take raw `Real` params.
- **Fix:** Either (a) integrate types into API — `d1_spring_elementstiffness(s::Spring{1})` alongside raw-param dispatch, or (b) move to `src/experimental/` with deprecation notice. Architect recommends (a) as it would validate parameters at construction time.

## HIGH (7 findings)

### H1. [NEW] Missing validation for E ≤ 0, G ≤ 0, I ≤ 0, J ≤ 0, k ≤ 0
- **File:** `src/beam.jl`, `src/truss.jl`, `src/spring.jl` (validation only for L)
- **Found by:** Defender (#8), Cross-Validator (new #2), Human Factors (#6)
- **Risk:** Zero/negative `E`, `I`, `G`, `J`, `k` silently produce zero/incorrect stiffness matrices. While `A ≤ 0` is intentional (parametric studies), `E = 0`, `k = 0` should warn. Spring with `k ≤ 0` is physically meaningless for structural analysis.
- **Fix:** Validate `E > 0`, `G > 0`, `I > 0`, `k > 0` in spring/beam functions with `ElementParameterError`.

### H2. [NEW] `_assemble!` doesn't validate element matrix size vs `ndofs`
- **File:** `src/assembly.jl:21-28`
- **Found by:** Prosecutor (#7)
- **Risk:** Passing 4×4 matrix with `ndofs=3` → silent wrong slices. Wrong DOF count causes cryptic `BoundsError`.
- **Fix:** `@assert size(k) == (2*ndofs, 2*ndofs) ElementParameterError(...)`.

### H3. [NEW] d3_beam rotation matrix copy-pasted across 2 functions (~25 lines each)
- **File:** `src/beam.jl:200-226` (stiffness), `:297-322` (forces); also `test/comparison.jl` has it again
- **Found by:** Architect (#3), Defender (#9), Cross-Validator (new #1)
- **Risk:** Bug fix in one copy will miss the other. Already diverged: stiffness uses `R' * kprime * R`, forces uses `kprime * R * u`.
- **Fix:** Extract `_d3_beam_rotation_matrix(x1,y1,z1,x2,y2,z2) -> (R, L)` helper. Apply threshold fix (C3) at the same time.

### H4. [PREV] Direction cosine computation repeated 15+ times across spring/truss/beam
- **File:** `src/spring.jl`, `src/truss.jl`, `src/beam.jl` (ubiquitous)
- **Found by:** Architect (#2), Cross-Validator (still valid #4)
- **Previous #4:** `deg2rad(theta)` + `cos`/`sin` → `Cx, Cy, Cz` repeated independently in every function. Inconsistent variable naming for same computation (see H5).
- **Fix:** Extract `_direction_cosines(theta...) → (Cx, Cy, Cz)` and/or `_transformation_matrix(theta...)` utilities.

### H5. [PREV] Inconsistent variable naming for same `thetay` computation in d3_truss
- **File:** `src/truss.jl:283` vs `:316,342,369`
- **Found by:** Cross-Validator (still valid #5)
- **Previous #5:** In stiffness: `u = deg2rad(thetay)`. In forces/strain/stress: `w = deg2rad(thetay)`. Same in `d3_spring`: `u` (stiffness) vs `uy` (force).
- **Fix:** H4 would eliminate this. Minimal fix: rename all to `θy`.

### H6. [NEW] d2/d3_beam_elementforces don't validate displacement vector length and other inputs
- **File:** `src/beam.jl:78-106`, `:278-325`
- **Found by:** Prosecutor (#9, #13)
- **Risk:** 5-element vector for d2 (expects 6) or 10-element for d3 (expects 12) → cryptic `BoundsError`. No check for numeric element type.
- **Fix:** `@assert length(u) == 6/12 "...")` with descriptive message.

### H7. [PREV] 3 custom error types defined, exported, never thrown
- **File:** `src/errors.jl`
- **Found by:** Human Factors (#3), Cross-Validator (still valid #1)
- **Previous #1:** `ElementDimensionError`, `AssemblyError`, `DiagramError` — defined with `showerror` methods, exported, never raised. Only `ElementParameterError` is used (12+ sites).
- **Fix:** Either (a) throw them in appropriate places (assembly errors, dimension mismatches, diagram failures) or (b) remove them. Add `@deprecate` if downstream code imports them.

## MEDIUM (12 findings)

### M1. [PREV] `BenchmarkTools` in both `[deps]` and `[extras]` in Project.toml
- **File:** `Project.toml:7,16`
- **Found by:** Architect (#5), Cross-Validator (still valid #3)
- **Impact:** Unnecessary precompilation cost for all users. Benchmarks run manually, not in CI.
- **Fix:** Remove `BenchmarkTools` from `[deps]`. Keep only in `[extras]`.

### M2. [PREV] `Plots` is a heavy dependency for 8 trivial line plots
- **File:** `Project.toml:8`, `src/plot.jl` (full), `src/LibFEM.jl:3`
- **Found by:** Architect (#6), Defender (#11), Cross-Validator (still valid #6)
- **Impact:** `using LibFEM` takes 30-60s due to Plots precompilation even if user only needs stiffness matrices.
- **Fix:** Make plots a `[weakdeps]` (Julia 1.9+) with lazy `import` in `plot.jl`. Or provide diagram data-only functions (already in `comparison.jl`) for headless use.

### M3. [PREV] `@kwdef` struct defaults allow silently invalid elements
- **File:** `src/types.jl:51-54,66-70,90-98`
- **Found by:** Cross-Validator (still valid #8)
- **Previous #8:** `Spring{2}()` compiles with `k=0.0, theta=0.0` — zero-stiffness 2D spring. No error. All struct fields default to 0.0 which is often meaningless.
- **Fix:** Add inner constructors with validation. Or mark with comment: "zero defaults for structural convenience; validate before use".

### M4. [NEW/PREV] `elementforce` vs `elementforces` naming inconsistency
- **File:** `src/spring.jl` (singular) vs `src/truss.jl` (plural)
- **Found by:** Human Factors (#2), Architect (#7)
- `d1_spring_elementforce` (returns 2-vector of nodal forces), `d2_spring_elementforce` (returns scalar axial force), `d1_truss_elementforces` (returns 2-vector of nodal forces), `d2_truss_elementforces` (returns scalar axial force)
- Fix: Standardize naming — `_elementforces` for nodal force vectors (returning per-node values), `_elementforce` for scalar axial forces.

### M5. [NEW] Diagram functions copy-paste pattern — 8 near-identical functions
- **File:** `src/plot.jl:25-202`
- **Found by:** Architect (#9)
- Each diagram function is 6-8 lines, differs only in title and f-index mapping. Pattern copy-pasted 8 times (3 for d2 + 5 for d3, plus the 8 data-only versions in `comparison.jl`).
- **Fix:** Generic `_beam_diagram(f, L, title, indices)` with index-pair mapping. Or use metaprogramming.

### M6. [PREV] `_assemble!` mutates **and** returns `K` (bang convention leak)
- **File:** `src/assembly.jl:28`
- **Found by:** Prosecutor (#8), Defender (#2), Architect (#8)
- **Previous nitpick #2:** `_assemble!` uses `.+=` on views (in-place) then `return K`. Works correctly but the `!` name suggests no-return mutation while all callers treat it as functional.
- **Fix:** Remove `return K`. Update callers: `d1_spring_assemble(K, k, i, j) = (_assemble!(K, k, i, j, 1); K)`. Or keep functional style and rename to `_assemble` (no `!`).

### M7. [PREV] Plots.Plot type test fragility in diagram tests
- **File:** `test/runtests.jl:191,198,205,591-596`
- **Found by:** Cross-Validator (still valid #4)
- **Previous nitpick #4:** `@test p isa Plots.Plot` ties tests to Plots.jl's internal type hierarchy. Upstream change → false failures.
- **Fix:** Structural test — `@test plottitle(p) == "Axial Force Diagram"` or check `p[:series]` length.

### M8. [PREV] Diagram docstrings don't explain force vector DOF layout
- **File:** `src/plot.jl` (all diagram docstrings)
- **Found by:** Cross-Validator (still valid #5)
- **Previous nitpick #5:** Docstrings say "6-element" / "12-element" but don't document which index maps to which DOF.
- **Fix:** Add to each docstring: `f[1]=Fx1, f[2]=Fy1, f[3]=Mz1, f[4]=Fx2, f[5]=Fy2, f[6]=Mz2` etc.

### M9. [NEW] Diagram functions don't validate `L > 0`
- **File:** `src/plot.jl` (all diagram functions)
- **Found by:** Defender (#10)
- **Fix:** `L > 0 || throw(ElementParameterError("L", ...))` before plotting.

### M10. [NEW] Plot functions don't work headless (CI failure without ENV setup)
- **File:** `src/plot.jl:25-202`
- **Found by:** Prosecutor (#11)
- **Fix:** Document `ENV["GKSwstype"] = "100"` for headless use, or lazy-init the backend.

### M11. [PREV] Missing `@test_throws` for `d2_beam_elementforces` L>0
- **File:** `test/runtests.jl:224-227` (only tests stiffness)
- **Found by:** Cross-Validator (still valid #3)
- **Previous nitpick #3:** `d2_beam_elementforces:79` throws for L≤0 but test only covers `elementstiffness`.
- **Fix:** Add `@test_throws ElementParameterError d2_beam_elementforces(1,1,1,0,0,zeros(6))`.

### M12. [NEW] d1_spring vs d2_spring `elementforce` — completely different semantics
- **File:** `src/spring.jl:32-34` vs `:96-101`
- **Found by:** Prosecutor (#10)
- d1 returns nodal force vector (2×1) from stiffness matrix × displacement. d2 returns scalar axial force from raw parameters. A user extending from 1D to 2D must rewrite their force computation entirely.
- **Fix:** See C5 — pick a single convention.

### M13. [NEW] README quickstart incomplete — stops at solve, no post-processing shown
- **File:** `README.md` (Quickstart section)
- **Found by:** Human Factors (#5)
- **Problem:** Quickstart shows stiffness → assemble → solve (`K[2:2,2:2] \ [10.0]`), but never shows how to recover element forces, stresses, or how BC reduction works. A student following along can't connect element stiffness to the solution without reading the element reference table and guessing.
- **Fix:** Extend quickstart with: (1) explain BC reduction step, (2) show element force recovery from solved displacements, (3) add a "full workflow" schematic.

### M14. [NEW] Beam diagrams plot a single line between end-values — pedagogically weak
- **File:** `src/plot.jl` (all 8 diagram functions)
- **Found by:** Human Factors (#7)
- **Problem:** Each diagram plots a straight line between two points (value at node 1, value at node 2). For beams, shear is constant (no distributed load → correct) but moment varies linearly — the diagram should show the actual linear variation with labels at key points. Students expecting textbook-style diagrams (with fill shading between diagram and axis) get a bare line.
- **Fix:** Add axis labels (x/L), moment diagram fill, and critical value annotations. Or at minimum document as "schematic only".
